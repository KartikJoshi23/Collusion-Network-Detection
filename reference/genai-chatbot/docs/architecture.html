03Agent Roster — every brain, every job

Each agent has a narrow contract: a JSON shape it must emit and a set of tools it may call. This separation is what makes the pipeline auditable.

01Router

always fires
Job
Classify the user query as sql, rag, hybrid, or clarify by looking at lexical cues, table-name mentions, and the presence of policy/governance language.
Tools
None — single LLM call, structured JSON output.
Bias
Strategic framings ("if X happened to Y", "exposure", "lock-in") tilt toward hybrid so policy facts are pulled even when user asks data-only.
Output: {"route": "sql|rag|hybrid|clarify", "rationale": "..."}

02Intent Classifier

always fires
Job
Extract structured intent: entities, ops, metrics, filters, time-range, and a list of ambiguities with severity scores.
Tools
Few-shot selector retrieves top-3 structurally-nearest goldens from eval/goldens.json (cosine on intent vectors, never on raw text).
Output
Drives the clarification gate, the planner, and the SQL/RAG agents downstream.
Output: {intent, entities[], ops[], ambiguities[], clarity_score}

03Clarification Agent

clarity < 0.7
Job
Pick the highest-severity ambiguity and ask one targeted question. Emits multiple-choice when enumerable, free-text otherwise.
Pause
LangGraph interrupt() halts the graph. The frontend shows chips. User reply re-enters as a new turn.
Cap
3 rounds, then escalates to best-guess with low-confidence badge.
Output: {question, options[], round, severity}

04Planner

hybrid only
Job
Decompose a hybrid question into ordered sub-questions, each tagged sql or rag. Sub-answers feed forward to the next sub-question.
Why
Prevents the model from flailing across SQL+RAG in a single prompt. Each sub-step has a single tool focus.
Output: {plan: [{type, question, depends_on}]}

05SQL Agent

sql + hybrid
Job
Generate, execute, and self-correct DuckDB SQL. Schema-aware via schema.yaml with 30 FK relationships including self-referential manager hierarchy and purpose: semantic hints.
Tools
list_tablesdescribe_tablesample_rowscount_rowsrun_sql
Loop
Up to 3 retries on syntax / runtime errors. Failure rows feed back into the next attempt as evidence.
Hard rules
No silent prorating of annual values. OpEx + CapEx must both be summed for "spend". Cross-department AI spend rolls up from all flagged depts.
Output: {query, rows_markdown, columns, row_count, attempts}

06RAG Agent

rag + hybrid
Job
Hybrid retrieval (dense + BM25 + RRF) → cross-encoder rerank → grounded synthesis with inline [doc.pdf::secN] citations.
Tools
hybrid_searchrerankfetch_full_chunkfetch_full_doc
Stack
nomic-embed-text-v1.5 (768-dim, CPU) · BM25 · RRF k=60 · BAAI/bge-reranker-base.
Contextual prefix
Each chunk is embedded WITH a 1–2 sentence "this section is about…" preamble generated at ingestion. Anthropic's contextual-retrieval technique.
Output: {answer, citations: [{chunk_id, raw_text}]}

07Hybrid Executor

hybrid only
Job
Run the planner's sub-questions in order, feeding each prior sub-answer into the next sub-question's prompt as "Prior sub-answer N" context.
Why
Sub-step 2 needs the IDs from sub-step 1's SQL. Prior version ran each sub-step blind and lost row-level continuity.
Output: {sub_results: [{type, question, answer, evidence}]}

08Synthesiser

hybrid only
Job
Merge SQL sub-answer rows with RAG sub-answer policy excerpts into one coherent response. Reconcile citations.
Hard rules
Row-level enumeration: never collapse N rows of SQL into a sentence; render every row. Arithmetic-completeness: when retention bonus = 30% × CTC, show both inputs and the result.
Output: {answer (markdown w/ row tables + inline citations)}

09Read-back

complex queries
Fires
≥ 1 join · any aggregation · TOP-N · ≥ 2 filters · any time range.
Job
Paraphrase the resolved interpretation in one sentence with concrete values: "To confirm: top 5 customers, tier='Tier 1' AND region='APAC', ranked by arr_inr_lakhs DESC. Run?"
Pause
Emits readback SSE event. Frontend shows Yes/Adjust chips. "Adjust" loops back to Intent Classifier.
Output: {readback, mentions, chips: ["Yes, run", "Adjust"]}

10Critic

always fires
Job
Score the draft answer against the evidence. Looks for missing rows, wrong joins, fuzzy timeframes, missing citations, arithmetic inconsistencies.
Tools
Read-only DB + retriever — can re-query to spot-check.
Verdict
pass · retry (loop back to SQL/RAG) · uncertain (escalate) · fail (escalate).
Output: {confidence: 0–1, reasons[], issues[], verdict}

11Cross-Validator

conf ∈ [0.5, 0.85] OR uncertain/retry/fail
Job
Re-derive the answer by a different method: subqueries instead of joins · alternate retrieval phrasing · alternate aggregation order. Compare against original.
Tools
Same toolkit as SQL + RAG agents; FactForge facts pre-loaded.
Special
On grounding-failure, emits a revised_answer with weave-in citations. Finaliser picks the higher-confidence verdict.
Output: {confidence, verdict, revised_answer?}

12Arbiter

rare · ~5%
Fires
Critic and Cross-Validator disagree by > 0.2 confidence OR conflicting verdicts.
Job
Run an independent third derivation from scratch with full DB + retriever access. Ties broken by majority on factual claims.
Output: {confidence, verdict, authoritative_answer?}

13Completeness Checker

post-validation
Job
Compare clarified-intent expectations ("show top 3 with retention bonus") against the rendered answer. Flags missing columns, missing arithmetic, dropped rows.
Reground
If a fact is referenced but not cited, runs another hybrid_search and weaves the citation back in.
Output: {complete, missing[], regrounded?}

14Finaliser

always fires
Job
Pick the authoritative draft (Cross-Validator's revision > Arbiter's verdict > original) and assemble the final SSE payload: answer + confidence + trace + evidence panel.
Output
The single final SSE event the frontend renders.
Output: {answer, confidence, trace, evidence: {sql, citations}}

04Data Layer

Three persistent stores plus a cold ingestion pipeline that runs once at build time.

DuckDB

embedded · 15 tables · ~1,800 rows
  • Reads .xlsx directly via read_excel_auto.
  • Schema normalised to snake_case.
  • Recursive CTEs for manager hierarchy.
  • 30 FK relationships in schema.yaml with purpose: semantic hints.

Qdrant

vector store · ~50 chunks
  • Section-based PDF chunking (regex on numbered headings).
  • Contextual prefix prepended before embedding.
  • nomic-embed-text-v1.5, 768-dim, CPU.
  • Hybrid query: dense + BM25 + RRF k=60.

Redis

session store · checkpoints
  • Stores LangGraph checkpoints for interrupt() resume.
  • Conversation history per thread_id.
  • 24h TTL.
  • Remapped to host port 16379.

Langfuse v2

observability · trace replay
  • Every agent call logged with input/output/duration.
  • Full trace tree per query, replayable.
  • Postgres-only (pinned to v2 — v3 needs ClickHouse).

05FactForge — extracted facts as a first-class index

Embeddings alone collapse on semantic-but-not-keyword matches. FactForge solves it by extracting structured facts at ingestion and injecting the relevant ones directly into agent prompts.

What it is

  • 898 facts total — 200 curated + 698 auto-extracted at ingestion.
  • Every fact has: id, statement, categories, origin (curated/auto), source (PDF + section or table + column), confidence.
  • De-duplication threshold: cosine 0.92 between fact embeddings.
  • Confidence gate: 0.8 minimum to be included in retrieval pool.

Category co-activation

_CATEGORY_EXPANSIONS = {
  "ipo_readiness": {
    "ipo_milestone", "ipo_timeline",
    "nrr", "customer_growth",
    "valuation", "certification"
  },
  "breach": {
    "breach_absorption",
    "cybersecurity_budget",
    "incident_baseline", "insurance"
  },
  ...
}

A query mentioning "IPO" pulls facts tagged with any of the co-activated categories — fixes narrow keyword matching.

Curated-first ordering

When a query matches both a curated fact and an auto-extracted fact, the curated one wins — they're hand-checked. Tie-break order:

  1. Highest category-overlap with query.
  2. Origin priority: curated > auto-pdf > auto-schema.
  3. Stable sort by fact.id for deterministic ordering.

06Validation Chain — defence in depth

Every answer passes through up to four gates. Cheap gates fire first; expensive ones only when needed.

Critic

Always fires

Score-and-reasons review against evidence. Verdict ∈ pass / retry / uncertain / fail.

Cross-Validator

conf ∈ [0.5, 0.85]
OR uncertain/retry/fail

Re-derive by a different method. Compare. Emits revised_answer on grounding fixes.

Arbiter

disagreement > 0.2

Independent third derivation with full corpus access. Tie-breaks at majority on factual claims.

Completeness

post-validation

Compare clarified-intent expectations vs final answer. Reground missing citations or flag.

Graceful degradation — if all gates fail, the Finaliser still emits an answer but with a low-confidence badge and an evidence panel showing raw rows + chunks, plus a "ask me a clarifying question" chip that loops back to clarification.

End-to-end wall-clock from POST /api/chat/message to final SSE event arriving. Times measured on a Windows workstation, CPU-only, against gpt-5.4.

5 – 9 s
Single table, no aggregation. Router → Intent → SQL → Critic → Finaliser.
Pure SQL · complex
15 – 35 s
3-4 joins, aggregations, window functions. Plus 1 clarification round + readback.
Pure RAG
8 – 18 s
Hybrid retrieval + rerank + grounded synthesis. First call adds embedder/reranker load (+20s cold).
Hybrid · medium
45 – 90 s
Planner + 2-3 sub-questions + Synthesiser + Critic. Typical for retention/IPO queries.
Hybrid · full chain
120 – 180 s
Cross-Validator + Completeness + reground. Observed on audit-IPO and oncall-burnout queries.
Cold start
+25 s
First query of a session — embedder + reranker model load + Qdrant warm-up.

Why 52 s? The query exercised the most expensive branch outside of full hybrid execution: clarification round + read-back + 3 SQL attempts + a Critic retry on grounding + Cross-Validator re-derivation + Completeness check. A simple SQL query without clarification would land at 5–9 s; a fully cached repeat of this exact query in the same session would land at ~30 s (no model cold-start, no clarification, fewer SQL attempts).
