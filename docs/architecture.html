CollusionGraph — one stack, two crimes: how it runs

A completed deep-learning system that screens money-laundering networks and bid-rigging cartels with one graph pipeline — and explains every alert it raises. Built, measured and shipped: 404 automated tests, honest numbers, a human always making the final call.

System Architecture

Top to bottom: data enters at stage 1 and a decision leaves at stage 10. Stages 1–7 run offline as the deep-learning batch pipeline; stages 8–10 are the always-on product. The whole thing runs on one machine with docker compose up.

Unified graph schema — one format for both domains

Parquet + DuckDB · validated on write

Every dataset becomes the same picture: typed nodes (accounts, wallets, firms, tenders) connected by typed edges (payments, awards, bids), with labels (illicit / licit / unknown) and time stamps. One schema is why one model family serves both crime domains — the project's core idea.

Time-safe by construction
  • models train on the past, are tested on the future
  • features only use information available at the cutoff
  • a dedicated leakage test-suite enforces this in CI
Honest labels
  • “unknown” nodes carry structure but never train the model
  • training labels are rolled back to what was knowable at the time

4

Calibrated ensemble — one comparable risk score per node

Each model's scores are first converted into honest probabilities on held-out data, then combined by weighted average — so a weak model can never drown out a strong one. (Skipping calibration collapses the ensemble; we keep that measured failure in the results as a warning.)

collusion is a team crime — group the suspects▼
5

Community grouping → ranked alert queue

254 financial + 223 procurement alerts · plus a wallet-level queue

The Leiden algorithm finds tightly-knit clusters; each cluster's member scores roll up into one alert. Overlapping alerts are removed top-down, oversized clusters are capped, and the queue is cut to a fixed review budget — quality is measured exactly there (precision in the top-k an analyst can actually read).

The Algorithms, Explained Simply

What each piece does and how it works — in everyday language.

Graph Neural Networks — the heart of the systemstage 3 · used everywhere risk is scored

What it does: scores every account or company by looking at who it deals with, not just what it looks like.

How it works: each node starts with a profile of numbers; in a few rounds every node blends its neighbours' profiles into its own, so a node inside a laundering ring ends up looking measurably different from a normal one.

GATv2 — attentionstage 3 · main model on the Bitcoin graph

What it does: learns which of a node's connections matter most and listens to those the hardest.

How it works: computes an importance weight for every connection before blending — and those weights double as a first clue when explaining an alert.

GraphSAGE & R-GCN — direction and relation typesstage 3 · financial / procurement

What it does: treats “money in” differently from “money out”, and “won a tender” differently from “bought from”.

How it works: each direction and each relationship type gets its own learned mixing rule instead of one shared pot.

Focal loss — learning from 2% positivesstage 3 · trains every deep model

What it does: stops the model from lazily calling everything “innocent” (98% correct, 100% useless).

How it works: easy, already-correct examples earn almost no credit during training, so effort concentrates on the rare suspicious cases.

Autoencoders (DOMINANT & GAE) — anomalies without labelsstage 3 · the unsupervised arm

What it does: flags unusual nodes even where nobody ever labelled a crime.

How it works: a network learns to compress and rebuild the graph; whatever rebuilds badly is unusual. (Honest finding: weaker than the supervised models on our data — reported, not hidden.)

Calibration + ensemble — merging models fairlystage 4

What it does: combines several models without letting a loud weak one drown out a quiet strong one.

How it works: each model's scores become honest probabilities first (checked on held-out data), then they're averaged. Skip that first step and the ensemble collapses — we measured it.

Leiden grouping + de-duplication — the short queuestage 5

What it does: turns hundreds of thousands of node scores into a few hundred group-shaped alerts.

How it works: a community-finding algorithm spots tightly-knit clusters; clusters become alerts; overlapping alerts are removed from the top down so no analyst reads the same case twice.

XGBoost baselines — the honesty yardstickstage 3 · every headline comparison

What it does: fields the strongest classic method (boosted decision trees) before any deep-learning claim is made.

How it works: hundreds of small decision trees, each correcting the last. On the Bitcoin data the trees still win — we report that openly, with a statistical significance test.

PGExplainer — “show me why”stage 6 · evidence for every top alert

What it does: answers “which few connections made the model raise this alert?”

How it works: a small helper network keeps only the links the prediction truly depends on — then every explanation is verified: remove those links and the alert should collapse; keep only them and it should stand.

Pattern matcher + official red flagsstage 6 · runs on every alert, no AI involved

What it does: checks each alert against nine known collusion shapes and cites the official FATF / OECD red flag each corresponds to.

How it works: hand-written graph rules search the alert's neighbourhood — evidence that stands even if you distrust the neural network entirely.

The Copilot's guardrails — an AI that can't make things upstage 8 · wrapped around every answer

What it does: lets investigators question the system in plain English without the usual AI risks.

How it works: the Copilot may only read published results; every number must literally appear in what it looked up; accusing phrases are rewritten automatically; a 24-question exam gates every release.

Statistical rigor — why the numbers can be trustedbehind every published result

What it does: makes sure no result is a lucky accident.

How it works: every headline model is retrained five times with different random starts and reported with its spread; comparisons are re-checked thousands of times on resampled data; every country and market takes a turn being fully held out to prove the system works on places it never saw.

AWS Cloud Architecture

The same three Docker containers, hosted as a product. Two separate planes — the always-on serving path and a scheduled deep-learning plane that exists only while it runs. No GPU is ever provisioned 24/7.

AWS CLOUD — SERVING PLANE (always on)
1

Edge — Route 53 · CloudFront · WAF

TLS everywhere · free tier: 1 TB delivery + 10M requests / month

Analysts need only a browser. DNS resolves the product domain; CloudFront serves the dashboard from cache worldwide and forwards /api/* to the backend; the WAF firewall rate-limits and filters when public.

2. static website / api traffic split▼
2

Hosting & compute

S3 — the dashboard
  • React build in a private bucket behind CloudFront
  • hashed, cache-friendly bundles
API + Copilot — two options, same images
  • Launch: Lambda containers + API Gateway — costs $0 while idle, scales to zero
  • Growth: ECS Fargate containers behind a load balancer, two availability zones, CPU autoscaling
Security & ops around it
  • Secrets Manager holds the one LLM key
  • serving role: read-only on artifacts (IAM)
  • CloudWatch logs + latency alarms · Budgets alert from day one
3. read-only access to published results▼
3

S3 artifact bucket — versioned results

Queues, dossiers and reports live under runs/<run_id>/. A new batch run lands as a new prefix; the API reloads its index; rolling back a bad run means pointing at yesterday's folder. The container registry (ECR) keeps the three images beside it.

DEEP-LEARNING PLANE (scheduled — exists only while it runs)
1

EventBridge schedule → rented GPU → publish → shut down

daily or weekly cadence · failure alarms via SNS · last good run keeps serving

On schedule, a spot GPU instance (or SageMaker job) spins up, runs the full pipeline — ingest, train the GNNs, score, group, explain — writes fresh artifacts to the S3 bucket above, and terminates. Training can also run for free on a local GPU or Colab and upload the same files. The expensive hardware exists for a few hours a week, never 24/7.
